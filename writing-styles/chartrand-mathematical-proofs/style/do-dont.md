# Do / Don't Checklist: Chartrand Mathematical Proofs

> Practical guardrails for writing in this voice.

## Do

- [ ] Use inclusive "we" as the default pronoun — the reader and writer are exploring together
- [ ] Define every technical term formally on first use, bolding the term being defined
- [ ] Provide motivation and context before introducing a new concept or definition
- [ ] Give informal intuition in everyday language before stating the formal version
- [ ] Present concrete, labeled examples with step-by-step solutions after definitions
- [ ] Include proof analysis or commentary after nontrivial proofs, explaining the reasoning choices
- [ ] Use short, punchy sentences for emphasis after longer expository passages
- [ ] Deploy parenthetical asides for gentle clarifications and editorial commentary
- [ ] Weave in historical context, cultural references, or biographical details to humanize the subject
- [ ] Use forward references ("as we will see in...") and backward references ("as we mentioned...") for coherence
- [ ] Acknowledge that the material is challenging while framing difficulty as surmountable
- [ ] State key concepts multiple ways — informally, then formally, then by example
- [ ] Use italics for emphasis and for labeling (Proof, Solution, Proof Analysis)
- [ ] Signal transitions explicitly: "We now turn to...", "Let's consider...", "We are now prepared to..."
- [ ] End sections with exercises that reinforce the concepts just presented
- [ ] Use the contrastive pattern when teaching conventions: show the wrong way, explain why, then show the right way
- [ ] Scale scaffolding to difficulty — add Proof Strategy (scratch work) and Proof Analysis (reflection) for hard proofs
- [ ] Show limitations of the current approach before introducing a new technique
- [ ] Use "Result" for most formal statements; reserve "Theorem" for especially important ones
- [ ] Narrate the thinking process for difficult proofs — show how choices emerge from informal reasoning

## Don't

- [ ] Skip the motivation step — never drop a definition without context
- [ ] Use undefined technical jargon — if you haven't defined it, you can't use it
- [ ] Write "obviously," "clearly," or "trivially" unless it truly is at the reader's level
- [ ] Default to passive voice in expository prose — "we prove" not "it is proved"
- [ ] Use bullet-point lists where flowing prose would serve better
- [ ] Rush through concepts — give each idea the space it needs to land
- [ ] Be condescending — the reader is intelligent but learning; respect that
- [ ] Overload with symbols — use prose to explain what the symbols mean
- [ ] Present mathematics as disconnected from human curiosity and discovery
- [ ] Use em-dashes where parentheses would fit the style better
- [ ] Introduce a proof technique without illustrating it with an example
- [ ] Let a proof stand alone without at least a brief remark on its strategy or significance
- [ ] Use "I" — this is always "we" (the authors and reader together)
- [ ] Write humor that distracts from the mathematical content — keep it subtle and purposeful
- [ ] Omit the proof-ending marker — always signal clearly where a proof concludes
