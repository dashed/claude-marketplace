# Ansible Playbook Authoring Field Guide

The practical reference for writing **ansible-core** playbooks: inventory, the
play→task→handler model, variables and their precedence, loops, conditionals,
Jinja2 templating, roles, collections, delegation, execution strategies,
handlers, tags, and check/diff/idempotency. For the CLI binaries, ad-hoc usage,
and the conceptual overview, see [SKILL.md](../SKILL.md). For `ansible.cfg`
settings and Ansible Vault, see [config-vault.md](config-vault.md).

> **Scope & version note:** This covers **ansible-core** (the engine shipping the
> `ansible.builtin` content set) — *not* the community `ansible` package bundle.
> Anything beyond `ansible.builtin` is a separate Galaxy collection. Playbook
> *keywords* (`hosts:`, `tasks:`, `loop:`, `when:`, `notify:`, …) carry **no
> per-keyword version metadata in the source** and are long-standing core, so
> they are left unannotated. Only features with a real maintainer-declared
> `version_added` are flagged inline as `(ansible-core X.Y+)`; the marker is the
> first release that shipped the feature. Documented against the ansible-core
> source at **2.22.0.dev0** (latest released line: **2.21**). Confirm names and
> options on your install with `ansible-doc <plugin>` and `ansible --version`.

## Inventory

Inventory declares the hosts you manage and groups them. The two hand-written
formats are **INI** and **YAML**; the default inventory path is
`/etc/ansible/hosts` (override with `-i` or `[defaults] inventory`).

**INI format** — groups in `[brackets]`, `:children` for group-of-groups,
`:vars` for group variables:

```ini
# inventory.ini
mail.example.com

[web]
web1.example.com
web2.example.com
www[01:50].example.com        # range expansion → www01 … www50

[db]
db1.example.com ansible_host=10.0.0.5 ansible_user=postgres

[prod:children]               # prod is a parent of web and db
web
db

[prod:vars]                   # vars applied to every host in prod
ansible_python_interpreter=/usr/bin/python3
```

**YAML format** — nested `children:` / `hosts:` / `vars:`:

```yaml
# inventory.yml
all:
  children:
    web:
      hosts:
        web1.example.com:
        web2.example.com:
    db:
      hosts:
        db1.example.com:
          ansible_host: 10.0.0.5
          ansible_user: postgres
    prod:
      children:
        web:
        db:
      vars:
        ansible_python_interpreter: /usr/bin/python3
```

Two groups always exist implicitly: **`all`** (every host) and **`ungrouped`**
(hosts in no other group).

### group_vars / host_vars

Keep variables out of the inventory file by placing them in `group_vars/` and
`host_vars/` directories. A file (or directory of files) named after the group
or host is auto-loaded:

```
group_vars/
  all.yml          # applies to every host
  web.yml          # applies to the `web` group
  prod/            # a directory is merged file-by-file
    secrets.yml
    tuning.yml
host_vars/
  web1.example.com.yml
```

These directories are resolved relative to **both** the inventory file's
directory **and** the playbook's directory. When both exist, the
**playbook-adjacent** copy has higher precedence (see
[Variables & Precedence](#variables--precedence)).

### Host patterns & `--limit`

The `hosts:` key (and `-l`/`--limit` on the CLI) accept patterns:

| Pattern | Meaning |
|---|---|
| `web` | the `web` group |
| `all` / `*` | every host |
| `web1.example.com` | a single host |
| `web:db` | union — hosts in `web` **or** `db` |
| `web:&prod` | intersection — in `web` **and** `prod` |
| `web:!web1.example.com` | exclusion — `web` except `web1` |
| `~web\d+\.example\.com` | `~` prefix = regex match |
| `web[0:2]` | index/slice of a group's hosts |

`-l`/`--limit` further restricts a run to a subset of the play's hosts (prefix
`@` to read the list from a file). `TRANSFORM_INVALID_GROUP_CHARS` defaults to
`never` — group names with invalid characters warn rather than being silently
mangled.

### Dynamic & plugin inventory

Inventory can be generated by **inventory plugins** instead of a static file.
The enabled set (`[inventory] enable_plugins` / `INVENTORY_ENABLED`) defaults to
`host_list, script, auto, yaml, ini, toml`; the full builtin set also includes
`advanced_host_list`, `constructed`, and `generator`. The **`auto`** plugin
dispatches a YAML inventory by its top-level `plugin:` key:

```yaml
# my_cloud.aws_ec2.yml — consumed by the `auto` plugin
plugin: amazon.aws.aws_ec2          # FQCN of an inventory plugin (a collection)
regions:
  - us-east-1
keyed_groups:
  - key: tags.Role
    prefix: role
```

Point `-i my_cloud.aws_ec2.yml` (or set it in `[defaults] inventory`) and
ansible runs the plugin to build the host list. Inspect the result with
`ansible-inventory -i <src> --graph` or `--list`.

## Playbook Anatomy

A **playbook** is an ordered list of **plays**. A play binds a host pattern
(`hosts:`) to an execution scope: variables, `roles`, and the task sections
`pre_tasks`, `tasks`, `post_tasks`, plus `handlers`.

```yaml
- name: Configure web tier            # ── a play ──
  hosts: web
  become: true                        # privilege escalation for the whole play
  gather_facts: true                  # run the setup module first (default)
  vars:
    http_port: 80
  pre_tasks:
    - name: Update apt cache
      ansible.builtin.apt:
        update_cache: true
  roles:
    - common
    - nginx
  tasks:
    - name: Deploy site config
      ansible.builtin.template:
        src: site.conf.j2
        dest: /etc/nginx/conf.d/site.conf
      notify: reload nginx            # fire a handler if this task changes
  post_tasks:
    - name: Smoke-test the endpoint
      ansible.builtin.uri:
        url: "http://localhost:{{ http_port }}/health"
  handlers:
    - name: reload nginx
      ansible.builtin.service:
        name: nginx
        state: reloaded
```

**Execution order within a play:** `pre_tasks` → (handlers notified so far) →
`roles` → `tasks` → (handlers notified so far) → `post_tasks` → (handlers).
Handlers run at the **end of the play** (or at an explicit `meta:
flush_handlers`), once each, no matter how many tasks notified them.

### Blocks: rescue / always

**`block:`** groups tasks and is Ansible's try/except/finally. `rescue:` runs
only if a task in the block fails; `always:` always runs. Directives on the
block (`when:`, `become:`, `tags:`, …) apply to every child task.

```yaml
- name: Resilient deploy
  block:
    - name: Pull new release
      ansible.builtin.git:
        repo: "{{ repo_url }}"
        dest: /srv/app
    - name: Restart service
      ansible.builtin.service:
        name: app
        state: restarted
  rescue:
    - name: Roll back on failure
      ansible.builtin.command: /srv/app/rollback.sh
  always:
    - name: Always record the attempt
      ansible.builtin.command: /usr/local/bin/log-deploy.sh
  when: deploy_enabled | bool
```

## Variables & Precedence

When the same variable is defined in more than one place, ansible-core merges
sources in a fixed order — **later sources override earlier ones**. The ordering
below is the **source-verified backbone** read directly from
`VariableManager.get_vars` (`lib/ansible/vars/manager.py`), lowest → highest:

| # | Source (low → high) | Where it lives |
|---|---|---|
| 1 | **role `defaults/`** *(weakest)* | `roles/<role>/defaults/main.yml` |
| 2 | inventory & group vars | inventory file/script group vars → `group_vars/` (inventory-adjacent, then playbook-adjacent); `all` group first, then child groups |
| 3 | host vars | inventory `host` vars → `host_vars/` (inventory-adjacent, then playbook-adjacent) |
| 4 | host facts / cached facts | gathered by `setup`, or `set_fact` with `cacheable: true` |
| 5 | play `vars:` | the play's `vars:` block |
| 6 | play `vars_files:` | files listed in the play's `vars_files:` |
| 7 | role (public/exported) vars | vars a role exposes to the rest of the play |
| 8 | role `vars/` then task `vars:` | `roles/<role>/vars/main.yml`, block vars, task `vars:` |
| 9 | `include_vars` | the `ansible.builtin.include_vars` module |
| 10 | `set_fact` / registered vars | `set_fact` (non-cacheable) and `register:` results |
| 11 | role params / include params | `roles: [{role: x, k: v}]`, `include_role`/`import_role` args |
| 12 | **extra vars (`-e`) — always wins** | `ansible-playbook -e key=val` / `-e @file.yml` |

The two rules to internalize: **`-e` extra vars override everything**, and role
**`defaults/` are the weakest** (designed to be overridden). For most authoring,
prefer setting values in `group_vars`/`host_vars` and role `defaults/`, reserving
`-e` for one-off overrides.

> ⚠ The official documentation expands this into a canonical **22-level** list
> (splitting inventory vs playbook `group_vars/all`, `group_vars/*`,
> `host_vars/*`, `vars_prompt`, block vars, etc.). That refinement comes from the
> docs, **not** verifiable from the engine source — the 12-step order above is
> the verified backbone and is correct about which source wins.

**Dict/hash merging:** `DEFAULT_HASH_BEHAVIOUR` (`[defaults] hash_behaviour`)
defaults to `replace` — a dict variable defined at a higher-precedence source
*replaces* the whole dict, it does not deep-merge. Setting it to `merge` is
discouraged (it is global and surprising); prefer
[`combine`](#templating-jinja2) in a template instead.

Common ways to set variables inside a play:

```yaml
- ansible.builtin.set_fact:           # define/compute a var at runtime
    app_version: "{{ git_tag | default('latest') }}"

- name: Capture command output
  ansible.builtin.command: id -un
  register: whoami_result             # whoami_result.stdout now usable

- name: Use the captured value
  ansible.builtin.debug:
    msg: "Running as {{ whoami_result.stdout }}"
```

## Loops

Prefer the modern **`loop:`** keyword. The default loop variable is **`item`**.

```yaml
- name: Install packages
  ansible.builtin.package:
    name: "{{ item }}"
    state: present
  loop:
    - git
    - curl
    - jq
```

### loop_control

`loop_control:` tunes loop behavior:

| Key | Purpose |
|---|---|
| `loop_var` | rename `item` (essential when nesting loops or including looped tasks) |
| `label` | control what each iteration prints (hide noisy/sensitive items) |
| `index_var` | bind a 0-based iteration index to a variable |
| `pause` | seconds to wait between iterations |
| `extended` | expose `ansible_loop` (first/last/index/length, …) |

```yaml
- name: Create users
  ansible.builtin.user:
    name: "{{ user.name }}"
    groups: "{{ user.groups }}"
  loop: "{{ users }}"
  loop_control:
    loop_var: user
    label: "{{ user.name }}"          # print just the name, not the whole dict
    index_var: idx
```

### until / retries / delay

Retry a task until a condition holds (a do-until loop, distinct from `loop:`):

```yaml
- name: Wait for the service to come up
  ansible.builtin.uri:
    url: http://localhost:8080/health
    status_code: 200
  register: health
  until: health.status == 200
  retries: 10
  delay: 6                            # seconds between attempts (default 5)
```

### Legacy `with_*`

The older `with_<lookup>:` loops still work but are discouraged in favor of
`loop:` (often combined with a filter or lookup):

| Legacy | Modern equivalent |
|---|---|
| `with_items: "{{ x }}"` | `loop: "{{ x }}"` (flattens one level — use `\| flatten` if needed) |
| `with_list: "{{ x }}"` | `loop: "{{ x }}"` |
| `with_dict: "{{ d }}"` | `loop: "{{ d \| dict2items }}"` |
| `with_fileglob: "*.conf"` | `loop: "{{ query('fileglob', '*.conf') }}"` |
| `with_together: [...]` | `loop: "{{ a \| zip(b) \| list }}"` |
| `with_nested: [a, b]` | `loop: "{{ a \| product(b) \| list }}"` |
| `with_sequence: start=1 end=5` | `loop: "{{ range(1, 6) \| list }}"` |

## Conditionals

`when:` takes a **raw Jinja2 expression** — do **not** wrap it in `{{ }}`:

```yaml
- name: Restart only on Debian family
  ansible.builtin.service:
    name: nginx
    state: restarted
  when: ansible_facts['os_family'] == "Debian"

# A list of conditions is implicitly AND-ed:
- ansible.builtin.command: /opt/migrate.sh
  when:
    - app_state == "running"
    - migrate | bool
```

**`changed_when:`** and **`failed_when:`** override how a task reports status —
indispensable for `command`/`shell`, which otherwise always report `changed`:

```yaml
- name: Check cluster health (never reports changed)
  ansible.builtin.command: clusterctl status
  register: cs
  changed_when: false
  failed_when: cs.rc != 0 or "DEGRADED" in cs.stdout
```

> Two strictness toggles arrived in **ansible-core 2.19**:
> `ALLOW_BROKEN_CONDITIONALS` and `ALLOW_EMBEDDED_TEMPLATES`. Leave them at
> their defaults — they relax checks that catch genuine authoring bugs.

## Templating (Jinja2)

Templates use Jinja2: `{{ expression }}` for substitution, `{% statement %}` for
control flow. Used inline in playbooks and in `.j2` files via the
`ansible.builtin.template` module.

**Common filters** (transform a value with `|`):

| Filter | Purpose |
|---|---|
| `default('x')` | fall back when undefined (`default('x', true)` also for empty/false) |
| `to_json` / `to_nice_json` | serialize to JSON (compact / indented) |
| `to_yaml` / `to_nice_yaml` | serialize to YAML |
| `regex_replace('p', 'r')` | regex substitution (filter gained `count`, `mandatory_count` in **ansible-core 2.17**) |
| `map('attribute', 'name')` | pull one field from each element |
| `select('match', 'p')` / `reject(...)` | keep / drop elements by a test |
| `combine(other)` | merge dicts (recursive with `recursive=true`) |
| `dict2items` / `items2dict` | convert between dict and list-of-pairs |
| `flatten` | collapse nested lists one or more levels |
| `b64encode` / `b64decode` | base64 (both gained `urlsafe` in **ansible-core 2.19**) |
| `commonpath` / `normpath` | path helpers (**new filters in ansible-core 2.15**) |

**Tests** (boolean checks with `is`):

```yaml
when:
  - my_var is defined
  - my_var is not none
  - some_version is version('2.0', '>=')   # version-comparison test
  - result is succeeded                     # also: failed, changed, skipped
```

Two tests added in **ansible-core 2.18**: `vaulted_file` (is a path
vault-encrypted) and `timedout` (did a task hit its `task_timeout`).

**Lookups vs `query`:** lookup plugins read external data **on the controller**.
`lookup('name', ...)` returns a (comma-joined) string by default;
`query('name', ...)` (alias `q`) always returns a **list** — prefer `query` when
feeding `loop:`.

```yaml
- ansible.builtin.debug:
    msg: "{{ lookup('ansible.builtin.env', 'HOME') }}"

- ansible.builtin.copy:
    content: "{{ lookup('ansible.builtin.file', item) }}"
    dest: "/etc/{{ item | basename }}"
  loop: "{{ query('ansible.builtin.fileglob', 'configs/*.conf') }}"
```

## Roles

A **role** is a reusable, self-contained bundle of tasks, handlers, vars,
templates, and files. Standard directory layout (each `main.yml` is the auto
entry point):

```
roles/nginx/
├── tasks/main.yml          # the role's tasks
├── handlers/main.yml       # handlers it can notify
├── defaults/main.yml       # variables — WEAKEST precedence (meant to override)
├── vars/main.yml           # variables — STRONG precedence
├── templates/              # .j2 files for the template module
├── files/                  # static files for copy/script
├── meta/main.yml           # metadata + role dependencies
├── library/                # role-local modules
└── module_utils/           # role-local module helper code
```

**`defaults/` vs `vars/`:** put user-tunable knobs in `defaults/` (low
precedence, easy to override from inventory/`-e`); reserve `vars/` for values the
role needs to pin and that should *not* be casually overridden.

**Dependencies** declared in `meta/main.yml` run before the role:

```yaml
# roles/nginx/meta/main.yml
dependencies:
  - role: common
  - role: certbot
    vars: { domains: ["example.com"] }
```

Three ways to consume a role:

| Mechanism | When it runs | Variable/handler scope |
|---|---|---|
| `roles:` keyword | at the role-execution phase of the play | role vars available play-wide |
| `import_role:` | **statically** at parse time (loop/conditional apply per-task inside) | `public: true` (**ansible-core 2.17**) exports the role's vars/handlers to the rest of the play |
| `include_role:` | **dynamically** at runtime (honors loops/conditionals on the include itself) | vars scoped to the include by default; `rescuable: true` (**ansible-core 2.21**) lets a failure inside be caught by an enclosing `block`/`rescue` |

```yaml
tasks:
  - name: Run a role conditionally at runtime
    ansible.builtin.include_role:
      name: nginx
    when: install_web | bool

  - name: Statically import and export its vars
    ansible.builtin.import_role:
      name: common
      public: true        # ansible-core 2.17+
```

Rule of thumb: **`import_*`** = static (decided at parse time, can't loop the
include itself); **`include_*`** = dynamic (decided at runtime, can loop and be
conditional).

## Collections & FQCN

Builtin content ships as **`ansible.builtin.<name>`** — always use the **fully
qualified collection name (FQCN)**: `ansible.builtin.copy`,
`ansible.builtin.service`, etc. Bare names (`copy`) resolve through the virtual
**`ansible.legacy`** namespace (local `library/` overrides, then builtin); FQCN
is unambiguous and the recommended style.

Anything outside `ansible.builtin` (e.g. `community.general`, `ansible.posix`)
is a **separate Galaxy collection** you must install. Declare dependencies in a
`requirements.yml`:

```yaml
# requirements.yml
collections:
  - name: community.general
    version: ">=8.0.0"
  - name: ansible.posix
  - name: https://github.com/org/repo.git    # git source
    type: git
    version: main
roles:
  - src: geerlingguy.docker
    version: "7.0.1"
```

```bash
ansible-galaxy collection install -r requirements.yml
ansible-galaxy collection install community.general          # single collection
ansible-galaxy collection install community.general -U       # upgrade
```

Reference collections in a play via the **`collections:`** keyword (lets you use
short names from those collections) — though explicit FQCN per task is clearer
and preferred:

```yaml
- hosts: all
  collections:
    - community.general
  tasks:
    - community.general.timezone:        # short name resolves via collections:
        name: Etc/UTC
```

Install location is controlled by **`COLLECTIONS_PATHS`** (`[defaults]
collections_path`, env **`ANSIBLE_COLLECTIONS_PATH`**), default
`~/.ansible/collections:/usr/share/ansible/collections`.

## Delegation & run_once

Run a task somewhere other than the current target host:

```yaml
- name: Add this host to the load balancer (run on the LB)
  community.general.haproxy:
    state: enabled
    host: "{{ inventory_hostname }}"
  delegate_to: lb1.example.com

- name: Gather facts onto the delegated host, not the target
  ansible.builtin.setup:
  delegate_to: db1.example.com
  delegate_facts: true

- name: Run a DB migration exactly once for the whole play
  ansible.builtin.command: /srv/app/migrate.sh
  run_once: true                        # executes on the first host only
```

`local_action:` is shorthand for `delegate_to: localhost` — useful for tasks
that act from the controller (sending a notification, calling an API):

```yaml
- name: Notify Slack from the controller
  local_action:
    module: ansible.builtin.uri
    url: "{{ slack_webhook }}"
    method: POST
    body: "{{ payload }}"
```

## Strategies & Parallelism

A **strategy** plugin decides how hosts advance through a play. Set it per play
(`strategy:`) or globally (`[defaults] strategy` / `DEFAULT_STRATEGY`, default
`linear`):

| Strategy | Behavior |
|---|---|
| `linear` (default) | lock-step — every host finishes a task before any host starts the next |
| `free` | each host races ahead through the play independently |
| `host_pinned` | each worker sticks with one host until it's done |
| `debug` | drops into the interactive task debugger on failure |

Tune parallelism with these play/CLI controls:

```yaml
- hosts: web
  serial: 2              # rolling batches of 2 hosts at a time (canary/rolling deploy)
  # serial: ["1", "30%", "100%"]   # escalating batch sizes
  throttle: 4            # cap concurrent execution of THIS task's hosts
  strategy: linear
```

`-f`/`--forks` (`[defaults] forks`, default **5**) sets how many hosts ansible
talks to in parallel overall; `serial` further restricts the *play* into rolling
batches; `throttle` caps a single task.

## Handlers

A **handler** is a task that runs only when **notified** by a changed task, at
the end of the play (or at `meta: flush_handlers`). Notifying the same handler
many times runs it **once**.

```yaml
tasks:
  - name: Deploy config
    ansible.builtin.template:
      src: app.conf.j2
      dest: /etc/app.conf
    notify: restart app           # only fires if the template CHANGED

handlers:
  - name: restart app
    ansible.builtin.service:
      name: app
      state: restarted
```

**`listen:`** lets several handlers subscribe to one notification topic — notify
the topic and every subscriber runs:

```yaml
tasks:
  - ansible.builtin.template:
      src: ca.crt.j2
      dest: /etc/ssl/ca.crt
    notify: "ssl certs changed"

handlers:
  - name: reload nginx
    ansible.builtin.service: { name: nginx, state: reloaded }
    listen: "ssl certs changed"
  - name: reload haproxy
    ansible.builtin.service: { name: haproxy, state: reloaded }
    listen: "ssl certs changed"
```

Force handlers to run mid-play with **`meta: flush_handlers`** (e.g. before a
dependent task), and run pending handlers even after a host failed with
**`--force-handlers`** (or `force_handlers: true`):

```yaml
- ansible.builtin.meta: flush_handlers
```

## Tags

Tag plays, blocks, tasks, and roles, then select or skip them with
`-t`/`--tags` and `--skip-tags`.

```yaml
tasks:
  - name: Install packages
    ansible.builtin.package: { name: nginx, state: present }
    tags: [install, packages]

  - name: Push config
    ansible.builtin.template: { src: nginx.conf.j2, dest: /etc/nginx/nginx.conf }
    tags: [config]
```

```bash
ansible-playbook site.yml --tags config         # run only config-tagged tasks
ansible-playbook site.yml --skip-tags packages  # run everything except packages
ansible-playbook site.yml --list-tags           # show all tags in the playbook
```

Reserved/special tag values:

| Tag | Effect |
|---|---|
| `always` | runs **unless** explicitly skipped with `--skip-tags always` |
| `never` | runs **only** when its tag (or `never`) is explicitly requested |
| `all` | (in `--tags`) every task — the default |
| `tagged` / `untagged` | select only tasks that have / lack any tag |

```yaml
- name: Dangerous reset — opt-in only
  ansible.builtin.command: /opt/wipe.sh
  tags: [never, wipe]        # runs only with --tags wipe
```

## Check Mode, Diff & Idempotency

**Idempotency** is the core principle: a well-written task reports `changed` only
when it actually changes the system, and re-running a converged playbook reports
zero changes. Prefer declarative modules (`copy`, `template`, `package`,
`service`) over `command`/`shell`; when you must use the latter, gate them with
`creates:`/`removes:` or `changed_when:`.

**Check mode** (`-C`/`--check`) is a dry run — it predicts changes without
applying them. **Diff** (`-D`/`--diff`) shows the textual change a task would
make (file contents, templated configs):

```bash
ansible-playbook site.yml --check --diff      # preview everything, change nothing
```

Control behavior per task:

```yaml
- name: Always run, even in --check (e.g. a read-only gather)
  ansible.builtin.command: /usr/local/bin/inventory-scan
  check_mode: false

- name: Force this task into check mode regardless of CLI
  ansible.builtin.lineinfile:
    path: /etc/hosts
    line: "10.0.0.1 db"
  check_mode: true
  diff: true
```

```yaml
- name: command made idempotent with creates:
  ansible.builtin.command: /opt/setup.sh
  args:
```
